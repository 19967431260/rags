# Playbook: 基于服务器数据判断用户是否真正加入音视频房间

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/9a1623846dc5425aa2b0e86549113dc0__音视频2-0基于服务器数据排查是否加入房间.md`
- scope: 呼叫组件场景、拿不到 SDK 日志时的服务端侧分诊
- intent: 判断问题停在“信令正常但未执行 joinChannel / getChannelInfo 失败 / join 后续阶段失败”哪一段

## 适用场景

适用于：
- 客户反馈一方已接听，但房间里只有一个人
- 看不到对方画面，且 QS 上 RTC 房间里只有一人
- 线上海量用户场景，拿不到组件或 NERtc SDK 日志

不适用于：
- 已确认双方都真正入房，只是听不到/看不到/卡顿

## 最小输入

至少要有：
- 问题时间点
- 主叫/被叫 IM uid
- 呼叫组件版本大致范围（1.8.0 前 / 1.8.0 后）

最好补齐：
- 信令 channelId
- RTC cid / join uid
- nrtc2_login 查询条件

## 首先判断

1. 这是旧版呼叫组件（1.8.0 前）还是新版（1.8.0 后）？
2. 信令层是否正常？
3. 是否实际发起了 `getChannelInfo`？
4. 是否有 `login` 事件上报？

## 分支排查路径

### A. 先确认信令逻辑是否正常

#### 1.8.0 以前旧版组件
典型流程：
- `15-1` 创建信令房间
- `15-4` 主叫加入信令房间
- `15-6` 主叫邀请被叫
- `15-9` 被叫同意
- `15-4` 被叫加入信令房间

判断：
- 若上述流程无报错，说明信令层面被叫大概率正常。

#### 1.8.0 之后新版组件
- 主叫 `15-16`
- 被叫 `15-17`

判断：
- 先通过 IM uid 查 `15-17`，拿到音视频 `cid` 和加入音视频的 `uid`。

### B. 确认是否有 `getChannelInfo` 成功记录

#### 旧版组件
- 可利用信令 `channelId` 搜 `-38-1`
- 老版本信令房间名规则较固定：`信令channelid | 音视频标记 | uid`

#### 新版组件
- 由于信令 channelId 命名不再固定，先从 `15-17` 取出 RTC `cid` 与 `uid`
- 再按 `cid + uid` 查 `-38-1`

判断：
- **没有 `-38-1`**：大概率连 `getChannelInfo` 都没走到，或前面就失败了。
- **有 `-38-1`**：说明至少开始进入 RTC join 前置链路。

### C. 确认是否有 `nrtc2_login` 上报

查询：
- TAG=`nrtc2_login`
- uid=目标查询 uid

关键理解：
- 客户端执行 `joinChannel` 时先 `getChannelInfo`
- 若 `getChannelInfo` 失败，也会上报 `login`
- 若成功，则继续 join，等后续有结果再上报 `login`

时间关系：
- 正常通话：`-38-1` 时间早，`login` 时间晚于 `getChannelInfo`

判断：
- **没有 `-38-1`，但有 `login` 且 channelid 异常**：偏 `getChannelInfo` 失败。
- **连 `login` 都没有**：要么根本没执行 `joinChannel`，要么在 `getChannelInfo` 后、建立媒体信令连接前就被杀进程/异常退出。

### D. 最终分诊

- 信令正常 + 无 `-38-1`：偏组件没触发到 RTC join，或 join 前失败
- 有 `-38-1` + login 异常：偏 getChannelInfo / join 早期失败
- 上述都正常：问题已进入后续媒体信令/ICE/媒体链路，转音频或视频手册继续排查

## 证据不足时必须补充

必须追补：
- 组件版本时代（1.8.0 前 / 后）
- 主被叫 IM uid
- 信令 channelId 或 `15-17` 中的 RTC `cid`、join uid
- 精确时间点
- 是否可能存在杀进程、切后台、异常退出

## 不要轻易下的结论

- 不要看到房间里只有一人，就直接认定 SDK join 失败；也可能根本没走到 `joinChannel`。
- 不要只因没有 `login` 就断言客户端没调用 join；也可能在中间阶段进程被杀或上报丢失。
- 不要把本 playbook 当作媒体质量排查工具；它只负责判断“是否真正执行到入房链路的哪一段”。

## 升级条件

- 组件链路、`-38-1`、`login` 三者关系仍无法解释现象
- 新旧版组件混用，日志命名不一致，无法可靠映射
- 已判断进入媒体连接阶段，需要转音频/视频/网络专项继续查
