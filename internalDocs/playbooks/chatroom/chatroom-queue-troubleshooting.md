# Playbook: 聊天室队列 / 麦位异常排查

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/77a629a9985d43ea9b19069a902358bd__聊天室队列问题排查.md`
- scope: 语聊房、聊天室麦位、队列元素增删、异常下麦

## 适用场景

适用于以下问题：
- 用户反馈“明明还在说话，但麦位掉了”。
- 用户反馈“人下麦了，但房间里还能听到他的声音”。
- 用户反馈“队列元素被系统自动清掉”。
- 用户怀疑客户端/服务端某次上麦、下麦操作异常。

## 最小输入

- roomid（聊天室 id）
- 问题用户 accid
- 问题发生时间点
- 麦位 key（如“麦位1”）和 value（如对应 accid）
- 操作方式：客户端接口还是服务端 API
- 是否使用 `transient=true`

## 首先判断

1. **先明确队列和 RTC 音频不是同一层能力**。
   - 队列表示“麦位状态/房间内逻辑座位”。
   - RTC 音频表示“是否仍在推流/订阅/听到声音”。
   - 所以“下麦后还能听到声音”并不自动等于队列异常，也不自动等于 RTC 异常，要两边分开看。

2. **先判断是不是 `transient=true` 导致的自动清理**。
   - 若提交者离开聊天室或掉线，且该元素是由其以 `transient=true` 添加，则元素会被系统删除。

3. **若 `transient=true` 且 operator 当时根本不在聊天室**：
   - 添加时应返回 403，先核实调用结果。

## 分支排查路径

### 路径 A：确认队列变更事实

1. 先通过 QS 查询聊天室历史消息。
2. 重点找**消息类型 5 的服务器通知消息**。
3. 在通知消息中重点关注：
   - `id=317`：主动操作引起的队列变更。
   - `id=320`：离线等系统触发的队列变更。
4. 从通知消息中提取：
   - `operator`：谁操作了队列。
   - `target`：谁是被操作对象。
   - `queueChange._e`：动作类型。
   - `queueChange.key` / `queueChange.content`：具体麦位键值。

### 路径 B：判断是主动操作还是系统清理

#### B1. 主动操作（317）
关注 `_e`：
- `OFFER`：添加队列元素（上麦/占位）
- `POLL`：取出队列元素（下麦）
- `DROP`：清空队列

然后去 Kibana 对照协议：
- `13-20`：SDK 新增队列元素
- `13-21`：SDK 取出队列元素
- `13-24`：SDK 清空队列元素
- `10015-11`：服务器新增队列元素
- `10015-12`：服务器清除队列元素
- `10015-14`：服务器清理队列元素

目标是确定：
- 谁发起了变更
- 是客户端还是服务端触发
- 该次操作是否成功

#### B2. 系统清理（320）
1. 重点看通知消息里是否出现 `PARTCLEAR`。
2. 若是：优先怀疑**用户离线后，系统按 transient 规则清理了其队列元素**。
3. 再去 Kibana 查该 accid 的登录登出状态：
   - 重点查对应时间点是否有 `13-5` 超时离线协议。
4. 如果离线证据和 320/`PARTCLEAR` 对得上，基本可以归因为离线触发的系统清理。

## 证据不足时必须补充

缺以下信息时，不要直接判断“异常下麦”：
- roomid
- 问题用户 accid
- 精确时间点
- 麦位 key/value
- 是否 `transient=true`
- QS 中 317/320 通知样本
- Kibana 中对应协议记录

## 不要轻易下的结论

- 不要把“还能听到声音”直接等同于队列没变。
- 不要把“队列被清理”直接等同于系统 bug，先排查 transient + 离线。
- 不要只看客户端现象，不查 317/320 通知消息。
- 不要忽略服务端 API 也可能改队列，不一定都是客户端 SDK 操作。

## 升级条件

- 317/320 通知、Kibana 协议、客户端表现三者明显矛盾。
- 无离线证据，却出现疑似系统自动清理。
- 调用应成功却无任何对应协议或通知下发。
- 房间队列状态正确，但 RTC 听感异常，需要转 RTC 链路排查。
