# Playbook: 推送问题标准排查（客户端取证版）

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/3e3fc5c9237b4b9e9ae0814ef408b7ee__推送问题排查V1-20220609.md`
- scope: iOS / Android / 服务端推送基础概念、标准排查顺序、常见配置错误

## 适用场景

适用于：
- 用户拿得到客户端 SDK 日志。
- 需要从 token 生成、云信上报、推送触发条件、证书/厂商通道配置一路做标准排查。
- 需要解释 payload、点击通知跳转、厂商限流等配套机制。

## 最小输入

### 通用
- appkey
- 发送方、接收方 accid
- 消息时间点、msgid
- 终端平台（iOS / Android）
- SDK 日志
- 客户端是否在线、是否前台、是否满足离线推送条件

### iOS 额外
- DeviceToken 文本
- 管理后台上传的 APNS 证书截图（名称、环境）
- App 打包环境（debug / release）
- 如做过外部工具验证，提供完整测试截图和结果

### Android 额外
- 厂商品牌、系统版本
- 厂商推送 token
- 推送证书/通道名称
- 通知栏权限、channel 开启情况

## 首先判断

1. **先确认是否满足推送前提**
   - 客户端必须按开发文档正确集成。
   - 若基础集成错误，后续日志排查意义很有限。

2. **再确认是否满足“应该触发推送”的时机**
   - 不是所有收消息场景都会推送。
   - 必须先核对是否符合离线推送条件、免打扰条件、会话与通知设置等。

3. **最后再区分是 token/证书问题，还是触发时机问题，还是厂商通道问题。**

## 分支排查路径

### 路径 A：iOS

#### A1. DeviceToken 是否生成
1. 查 `didRegisterForRemoteNotificationsWithDeviceToken` 是否正常回调。
2. 若没有回调：
   - 优先排查 iOS 推送配置、APNS 注册逻辑、系统权限。
   - 先不要去看云信后台，因为 token 都还没生成。

#### A2. Token 是否上报云信
1. 看 SDK 日志中的 token 相关打印。
2. 若日志显示 SDK 已获取并上报 token：
   - 说明问题不在“本地完全没 token”。
3. 若没有对应日志：
   - 继续回到客户端集成/注册链路排查。

#### A3. 是否满足推送条件
1. 对照“关于消息推送”的触发条件。
2. 若接收端其实在线、前台、或不满足离线推送条件：
   - 不要误判为推送失败。

#### A4. 后台证书配置是否正确
1. 核对：
   - 证书是否上传
   - 证书环境是否正确
   - app 打包环境与证书环境是否匹配
2. 典型错误：
   - 线上包却配测试环境证书
   - token 和 APNS 环境不匹配

#### A5. 用外部工具验证证书/token 有效性（若可行）
1. 可借助第三方 APNS 推送工具验证。
2. 工具验证成功但云信推送失败：
   - 重点回头核对：
     - 提供给外部工具的 token 是否与云信日志中的 token 一致
     - 云信后台上传的证书环境是否与测试时选择的环境一致
3. 注意：工具名称和可用性可能随时间变化，不要把某个具体工具当成唯一方案。

#### A6. 仍无法定位时
提交后端排查，至少附上：
- SDK 日志
- DeviceToken 文本
- 证书配置截图
- 外部工具测试截图
- 测试时间点和 msgid

### 路径 B：Android

#### B1. 厂商 token 是否生成
1. 查 SDK 日志中 `mix_push` 相关字段。
2. 若注册成功：说明厂商 token 已拿到。
3. 若注册失败：
   - 日志通常会带厂家 SDK 回调错误。
   - 先按厂家错误信息定位，而不是先怀疑云信服务器。

#### B2. token 是否上报云信
文档经验口径：
- 一般 token 成功生成后，只要 IM 登录成功，token 通常也会正常上报到云信。

#### B3. 是否满足推送条件
1. 继续核对离线推送触发条件。
2. 再检查通知栏权限是否开启。
3. Android 8.0+ 还要特别检查 channel 是否开启。

#### B4. 若都正常仍收不到
提交：
- token
- SDK 日志
- 证书名称 / 厂商通道信息
- 测试时间点与消息样本

### 路径 C：服务端/配置附加问题

#### C1. 服务端推送过期
- 若超过过期时间仍未送达，可能出现 `notification expire`。

#### C2. iOS 特定错误
- 若出现 `ex=java,io.IOException: Stream closed before a reply was received`，优先怀疑：
  - 线上应用使用了测试环境证书
  - 管理后台上传时选择了错误环境

#### C3. payload / 点击通知跳转问题
- 如果用户说“推送到了但点击后没跳转对”：
  - 不要按“收不到推送”排查。
  - 应转为 payload 字段、平台字段、客户端跳转接收组件是否匹配的问题。

#### C4. 厂商限流
- 若出现“偶发收不到、量大时更明显”：
  - 考虑厂商限流，而不是只怀疑云信。
- 常见方向：
  - 小米：普通级别消息限流，需申请更高等级并传 `channel_id`
  - 华为：单设备单应用有日限额
  - OPPO：默认公信通道限量，可申请私信通道
  - vivo：运营消息默认限条数，可用系统消息分类并申请增量

## 证据不足时必须补充

- 平台类型（iOS/Android）
- SDK 日志
- 测试时间点与 msgid
- token / DeviceToken
- 证书截图或厂商通道配置
- 是否满足推送时机
- 是否开启通知权限 / channel

## 不要轻易下的结论

- 不要一上来就说“云信没推”，先确认是不是本地根本没 token。
- 不要把“点击跳转不对”误判为“推送没到”。
- 不要忽略厂商限流、通知权限、channel 配置。
- 不要把第三方测试工具结果当成云信链路全量证明，它只能证明 APNS/证书/token 的一部分链路。

## 升级条件

- token 已正常生成并上报，证书/权限/条件都核对无误，仍无法收到推送。
- 需要后端查询实际推送请求与返回。
- iOS/APNS 或厂商侧返回异常码需要进一步结合服务端记录解释。
