# Playbook：漫游问题排查

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/a32a8c9a8dbb4fc9a9eb1306a8c58208__漫游问题排查.md`
- 定位：**完整度较高的漫游排查 playbook**。适合作为“换端后历史消息缺失 / 离线后登录未补消息”的主流程。

## 适用场景

- 用户换设备登录后，看不到另一设备已收到的历史消息
- 用户离线期间有人发消息，重新登录后未补到
- 已确认消息发送成功，但怀疑未正确入漫游库或未正确同步到新端

## 最小输入

至少要有：
- appkey
- 丢失漫游的一方账号（很重要）
- 对端账号 / 会话 ID
- 问题消息的大致时间范围

强烈建议补充：
- 消息ID / clientId / serverId
- 客户端日志（尤其登录阶段）
- 是否公有云 / 私有化
- 是否有撤回、删除漫游、单向删除等动作

## 首先判断

按这个顺序走：
1. **应用级漫游开关是否打开**
2. **消息本身是否有存漫游资格**
3. **QS 查询该会话漫游记录是否存在**
4. **若 QS 有漫游，客户端是否通过 5-1 / 4-9 正常同步并落库/展示**

注意：
- QS 查询时，**发送方一定要使用“丢失漫游的一方”**，否则结果可能不准。

---

## 背景约束

默认公有云漫游能力：
- 最近 **7 天**
- 最近 **100 个会话**
- 每会话最近 **100 条消息**

结论含义：
- 超过窗口的消息，不属于异常
- 公有云通常不支持直接改配置；若业务强依赖更高档位，需通过商务沟通专属云方案

---

## 分支排查路径

### A. 先确认漫游开关

#### A1. 应用级漫游开关
- 用 QS，传入 `appkey` 查询应用级漫游是否开启

若未开启：
- 直接说明根因：应用侧未启用漫游，后续消息不会进入漫游能力闭环
- 不必继续深挖单条消息

#### A2. 消息级漫游开关
- 在应用级开关开启后，继续确认问题消息自身是否设置“存漫游”
- 可通过消息 ID 查 Kibana；若无消息 ID，则按发送方 UID + 时间点查发送协议与消息对象属性

若消息明确未存漫游：
- 优先解释为业务方发送参数导致
- 如果不是用户有意关闭，可建议核对发送消息配置参数，去掉不存漫游设置

### B. QS 查询漫游不存在

若应用开关没问题，但 QS 查不到漫游记录，继续按下列顺序排：

#### B1. 是否超过漫游限制
检查是否超过：
- 7 天
- 100 会话
- 每会话 100 条

若超过：
- 属于能力边界，不是故障

#### B2. 发送时未存漫游
证据：
- 发送协议或消息对象中的漫游属性

若确认未存漫游：
- 说明这是配置/业务场景结果，不是服务器随机丢失

#### B3. 是否被主动删除或被撤回
检查：
- 是否调用过删除漫游接口
- 是否发生撤回
- 若用户同时存云端历史，可看该消息能否被云端历史单条查询到

判断：
- 云端历史也查不到，再结合 Kibana，可进一步支持“被撤回/被删除”的方向

#### B4. 是否未入漫游库或入库过慢
检查：
- 用 `sid:-36 cid:1` 查询该消息入漫游库时间
- 比较“入库时间”与“客户端登录同步时间”

典型结论：
- 如果客户端登录早于入漫游库完成，可能出现这次登录没有补到的现象

#### B5. 是否为服务器侧存储异常
前提：
- 没超限制
- 消息设置了存漫游
- 没有撤回/删除
- 但 QS 就是查不到某条应存在的漫游

结论：
- 高度怀疑服务器存储漫游侧 bug，应联系服务器同事进一步排查

### C. QS 查询漫游存在：转客户端同步链路分析

这类场景说明：
- 漫游库本身有数据
- 问题可能发生在“同步时点、下发、监听、入库、过滤、展示”任一环节

#### C1. 看 5-1：客户端上报的漫游时间戳是否合理
- 登录完成后，客户端会发 **5-1** 同步请求
- Android 日志通常直接打印漫游时间戳
- iOS 需看一组 tag，其中**tag7** 对应漫游时间戳

判断：
- 这个时间戳通常应接近“客户端实际收到的最后一条消息时间”
- 若明显不对，可能是版本 bug，或客户端插入消息/异常时间戳把漫游位置顶掉了

#### C2. 看 4-9：服务器是否下发了应有会话的漫游数据
- 服务器根据 5-1 时间戳，计算需同步的漫游会话，再通过 **4-9** 下发
- 每个会话的漫游数据通常对应一个 4-9

判断：
- **有 4-9** → 继续看客户端是否消费成功
- **应有数据但没有 4-9** → 倾向服务器问题；私有化环境更常见

#### C3. 看消息监听、本地历史、过滤器和数据库
在 4-9 正常的前提下继续判断：

- **日志里消息监听收到了，用户本地历史也有**
  - 更像 UI 渲染问题
- **日志里消息监听收到了，但本地历史没有**
  - 先看是否被过滤器过滤
  - 若无过滤器，怀疑数据库写入失败，或调用过本地删除消息接口
- **数据库写入失败**
  - 需要客户端开发继续排查

## 证据不足时必须补充

- appkey
- 丢失漫游的一方账号
- 对端账号 / 会话 ID
- 精确时间范围
- 消息 ID 或可检索标识
- QS 漫游查询结果截图或结果描述
- 登录阶段客户端日志（5-1 / 4-9 / 消息监听）
- 是否有撤回、删除漫游、单向删除、本地删消息
- 公有云还是私有化

## 不要轻易下的结论

- 不要把“QS 查不到漫游”直接等同于服务器 bug，先排限制、开关、未存漫游、删除/撤回、入库延迟
- 不要把“QS 有漫游”直接等同于客户端一定能看到，后面还有 5-1 / 4-9 / 监听 / DB / UI 链路
- 不要忘记用“丢失漫游的一方”做查询主体，否则结论可能失真
- 不要把超出 7 天 / 100 会话 / 100 条的情况当故障

## 升级条件

- 确认未超限制、消息设置了存漫游、无撤回/删除，但 QS 查不到应有数据
- 5-1 时间戳合理，漫游库也有应同步数据，但无 4-9 下发
- 私有化环境中反复出现“应下发未下发”现象
- 客户端已确认无过滤器/无本地删除，仍怀疑 SDK 入库失败或数据库异常
