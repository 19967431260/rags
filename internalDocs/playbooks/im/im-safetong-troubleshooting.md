# Playbook: IM 安全通拦截/放行异常排查

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/27b33e4ce67a4be68d4785905f54dc25__IM-安全通问题排查.md`
- scope: IM 文本/图片为主，补充涉及语音/视频异步安全通结果的处理口径

## 适用场景

适用于以下投诉：
- 用户说“明明配置了拦截，但消息还是发出去了”。
- 用户说“明明配置了疑似处理，但没有按预期拦/放”。
- 用户怀疑安全通检测超时导致消息放行。
- 用户问语音/视频为什么看起来命中了却没有自动撤回。

## 最小输入

至少收集：
- appkey
- 发送时间点
- 会话类型（单聊/群聊/聊天室/圈组）
- 消息类型（文本/图片/语音/视频）
- 发送方 accid、接收方/群 id/聊天室 id
- 1 条可复现样本或 msgid/taskid（能拿到 taskid 最好）
- 用户在控制台的策略配置截图（尤其是疑似处理配置）

## 首先判断

1. **先判消息类型是同步还是异步检测**
   - 文本、图片：通常按同步检测思路排查。
   - 语音、视频：按异步检测思路排查，不能用同步“是否立即拦截”来判断。

2. **先判云信侧是否识别为命中**
   - 查 Kibana / 相关日志中的 `antiSpam`。
   - `antiSpam=true`：说明云信侧已判定命中。
   - `antiSpam=false`：说明云信侧未判定命中，不要先下结论说“配置失效”。

3. **再判命中来源**
   - `antiSpam.type=normal`：通用反垃圾命中。
   - `antiSpam.type=yidun`：易盾反垃圾命中。

## 分支排查路径

### 路径 A：用户说“命中拦截策略，但没拦截”

1. 查是否真的命中：
   - 若 `antiSpam` 不为 `true`：
     - 结论不是“云信放过了命中的消息”，而是“云信侧未识别为命中”。
     - 此时需要取 `taskid` 交由易盾侧核查检测结果与策略命中情况。

2. 若 `antiSpam=true`：
   - 原则上同步消息不应正常放行。
   - 先继续查是否存在**检测超时后按控制台策略放行**。

3. 查 `antiSpam.spendMs`：
   - 文本重点关注是否超过 **500ms**。
   - 图片重点关注是否超过 **2s**。
   - 若超过阈值：优先判断为**检测超时 + 云信按“送检失败处理策略”执行**，不宜直接认定为规则未命中。

4. 若已命中且无明显超时仍放行：
   - 升级内部研发/专家排查，重点说明“已确认 antiSpam=true，但行为与预期不符”。

### 路径 B：用户说“命中疑似策略，但没拦截/没按疑似配置处理”

1. 先确认用户是否在**正确位置**勾选了疑似处理配置。
2. 查云信侧是否命中：
   - 若 `antiSpam=false`：
     - 获取 `taskid` 给易盾侧核查。
     - 重点问：当前是否命中了自定义 `businessId`；该 `businessId` 是否具备对应的疑似处理配置。
3. 若用户使用历史创建的自定义业务 ID：
   - 需特别怀疑该业务 ID 的疑似处理配置并未在当前控制台完整生效，需由易盾侧确认。

### 路径 C：语音/视频消息相关

1. 语音/视频是**异步安全通**。
2. 云信可能先行下发消息，再由易盾异步抄送结果。
3. 当前口径下，只有**命中**时才会有异步抄送。
4. 若异步结果显示命中：
   - 通常需要客户业务侧自行做撤回/处置。
   - 不要默认认为平台会自动完成所有后处理。

## 证据不足时必须补充

缺少以下任一项时，不要贸然归因：
- taskid
- 具体消息类型
- 控制台策略截图
- 发送时间点
- antiSpam / antiSpam.type / antiSpam.spendMs 对应日志
- 是否自定义 businessId

## 不要轻易下的结论

- 不要仅凭“用户认为命中”就判断一定命中。
- 不要把“疑似策略未生效”和“删除级命中”混为一谈。
- 不要忽略超时放行机制。
- 不要对语音/视频直接套用文本/图片的同步拦截结论。
- 不要在没有 taskid 时就断言易盾策略异常。

## 升级条件

出现以下情况应升级：
- `antiSpam=true` 且无明显超时，但消息仍异常放行。
- 用户使用自定义 businessId，怀疑配置绑定或历史迁移问题。
- 语音/视频异步抄送结果与客户接收行为严重不一致。
- 需要易盾确认 taskid 对应的实际检测结果与策略命中细节。
