# Playbook: 推送问题快速反查（仅 appkey / accid 也能启动）

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/40e8cfcbbe7f4565a8bb7255da0fc821__推送排查-内部版.md`
- scope: 缺少客户端 SDK 日志时，从服务端记录反推 token、证书名、是否尝试投递、为什么跳过

## 适用场景

适用于：
- 用户拿不到客户端 SDK 日志。
- 只能提供 appkey、accid、时间点，想先做一轮快速定位。
- 需要确认服务端是否拿到 token、使用了哪个证书名、是否真的尝试向 APNS/厂商投递。

## 最小输入

- appkey
- accid / uid
- 问题发生时间点
- 平台（iOS / Android，若未知也可先反查）
- 若有消息样本，补充 msgid / 会话类型更好

## 首先判断

1. **先从登录协议反查“设备上报了什么”**，而不是先猜客户端配置。
2. **再从推送日志确认“服务器是否尝试投递”**。
3. **最后再判断是没 token、证书名不匹配、在线跳过、免打扰跳过，还是已投递到第三方但第三方结果不可见。**

## 分支排查路径

### 路径 A：先查登录协议，拿 token 和证书名

#### A1. Android
- 通过 uid 查 `3-1` 协议。
- 重点字段：
  - `tokenName`：推送证书/通道名
  - `pushToken`：推送 token

#### A2. iOS
- 通过 uid 查 `2-2` 协议。
- 重点字段：
  - `logintag[10]`：推送证书名
  - `logintag[11]`：推送 token

#### A3. 若登录协议里拿不到 token
- 初步归因为“客户端登录时未成功上报推送信息”。
- 此时不要直接查深层推送失败原因，因为服务器侧可能根本没有可投递目标。

### 路径 B：根据 token 查推送记录

1. 若拿到了 token：
   - 直接按 token 查推送记录。
2. 若拿不到 token，但拿到了 appid + 推送证书名：
   - 退而查该应用该证书下整体推送情况。

### 路径 C：判断是否是“整个证书都在失败”

1. 如果整个应用/证书下都有成功记录：
   - 说明不是全局证书故障。
   - 更可能是**该用户当时不满足推送时机**、在线被跳过、token 异常等单点问题。
2. 如果整个应用/证书下都失败：
   - 重点查失败记录的 `desc`。
   - 常见可定位到“管理后台没有配置该名字的推送证书”等证书名不匹配问题。
3. 如需进一步确认后台已配哪些证书：
   - 可从 QS 侧查询该 appkey/appid 下配置的证书列表。

### 路径 D：看新推送日志，判断是否尝试投递第三方厂商

查询 `sid:-4 cid:1` 相关记录。

#### D1. `success=true`
- 说明 IM 服务器**已经尝试投递到第三方厂商 / APNS**。
- 但该日志**不包含第三方最终返回结果**。
- 所以不能据此断言“厂商一定推送成功到设备”。

#### D2. `success=false`
- 说明服务器没有实际投递，并会带理由。
- 常见原因包括：
  - `missing item: token`：没有 token
  - `team silent`：群免打扰
  - `skipDevicesNum > 0`：接收方在线设备被跳过推送

#### D3. `skipDevicesNum`
- 表示因在线而跳过推送的设备数。
- 若该值大于 0，优先考虑“用户当时在线，不属于异常不推”。

### 路径 E：免打扰判断

- 通过 QS 查对应消息的 `isMuted` 字段。
- 若 `isMuted=true`：说明接收方对发送方设置了免打扰，因此无法触发推送。

### 路径 F：工具与补充验证

文档中提到可通过内部工具下载证书、做定向推送验证，但 agent 转译时建议保留为方法论：
- 如需进一步验证证书本身可用性，可获取后台上传证书并做独立推送测试。
- 工具名和使用方式可能变化，不应把某个 jar 包名写成固定 SOP。

## 证据不足时必须补充

- appkey
- accid / uid
- 问题时间点
- 平台类型（若未知，可先反查）
- 登录协议样本
- 推送日志样本
- 是否单聊 / 群聊 / 是否存在免打扰

## 不要轻易下的结论

- 不要查到 `success=true` 就说“厂商一定推到了设备”。
- 不要查到没有推送记录就直接认定服务器没发，先看是否在线跳过或没有 token。
- 不要把“整个证书失败”和“单用户 token 失败”混成同一种问题。
- 不要忽略 `isMuted`、`team silent` 这类业务配置原因。

## 升级条件

- 登录协议显示已有 token，但推送记录和行为严重矛盾。
- 已尝试投递第三方，但终端始终无法收到，需要结合厂商/APNS 侧进一步排查。
- 证书名、证书配置、token 上报链路存在疑似系统级异常。
