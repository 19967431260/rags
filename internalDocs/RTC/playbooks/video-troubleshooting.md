# Playbook: RTC 视频问题排查

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/086ae672424f4b039d5c73d7769db8e2__视频问题排查手册.md`
- scope: RTC 实时视频/屏幕共享
- intent: 用发送端-网络-接收端-渲染链路，把“没画面/模糊/卡顿/音画不同步/曝光异常”切到正确责任段

## 适用场景

适用于：
- 看不到远端画面 / 本地预览没画面 / 远端也看不到
- 模糊、马赛克、绿块、首帧模糊
- 视频卡顿、掉帧
- 音画不同步
- 曝光、偏黑、旋转、屏幕共享清晰度问题

不适用于：
- 纯入房失败
- 纯服务端 API 报错

## 最小输入

至少要有：
- room id / uid / 问题时间点
- 谁看谁有问题
- 平台 / 系统版本 / SDK 版本
- QS 可查询时间窗

最好补齐：
- 是否本地预览异常
- 是否所有观众都异常还是单个接收端异常
- 是否订阅大流/小流
- 是否有前处理：美颜、虚拟背景、鉴黄、第三方渲染
- 是否屏幕共享
- 对应截图/录屏/拉流地址

## 首先判断

1. **先确认是否真的加入房间成功。**
2. **分清“发送端问题”还是“接收端问题”。**
   - A 自看也异常：更偏发送端/前处理/采集。
   - A 自看正常，仅 B 看异常：更偏编码/网络/订阅/解码/渲染。
3. **确认是大流还是小流。** 小流略模糊可能符合预期，不能直接当故障。

## 分支排查路径

### A. 没画面

顺序固定：
1. 确认入房成功
2. 看是否有发送码率：`Video Send Bitrate - Real`
3. 没发送时，看：
   - `Video Capture Framerate`
   - `Video Send Framerate`
   - `enableLocalVideo`
   - `setLocalVideoConfig`
4. 有发送无接收时：查服务器/传输链路
5. 有接收时：看
   - `Video Decode Out Framerate`
   - `Video Render Out Framerate`
   - `setupRemoteVideoCanvas`

高频结论：
- `Render Out Framerate = 0`：优先怀疑未设置远端画布或渲染链路异常。
- 本地预览无画面且远端也无：优先查摄像头采集、设备占用、权限、前处理。
- 远端无画面但本地预览正常：优先查编码/发送/订阅/接收/渲染。

### B. 模糊 / 马赛克 / 绿块

先分 3 类：
1. **绿块/码流错误**：直接偏视频编解码异常，尽快升级视频研发。
2. **小流模糊**：先确认是否订阅了小流；若是，轻微模糊可接受。
3. **大流模糊**：看
   - `Video Max Encode QP - High`
   - `Video Send Bitrate - Real`
   - 发送端分辨率 / 接收端分辨率

判断：
- QP < 35 但仍模糊：偏视频侧质量问题。
- QP 很大且码率明显偏低：先看网络/VQC/编码耗时。
- 主播自看良好、观众看模糊：先拿拉流地址或下行数据验证真实效果。

注意：文档明确提醒，不要把“直播内容质量差（距离远、没对焦、光线过曝、美颜过重）”误判为 SDK 问题。

### C. 画面卡顿

按接收端体验倒推：
1. 先确认卡顿真实发生，核对
   - `Video Decode Out Low Stuck Duration/Count`
   - `Video Decode Out High Stuck Duration/Count`
2. 看发送端采集：`Video Capture Framerate`
3. 看发送端编码输出：`Video Send Framerate`
4. 看接收：`Video Recv Framerate`
5. 看解码：`Video Decode Out Framerate`
6. 看渲染：`Video Render Out Framerate`
7. 看 `CPU/Mem`、订阅切换、频繁进出房间

判断：
- Capture 低：采集/前处理/设备性能
- Send 低：编码/VQC/客户端
- Recv 低：网络/服务器
- Decode 低：解码侧
- Render 低：渲染侧
- CPU > 90%：设备性能是高优先级嫌疑

### D. 音画不同步

优先看：
- `Video JB Target Delay`
- `Audio Sync Relative Delay`
- 发送端和接收端的帧率是否异常

经验阈值：
- 若延时 > 500ms，可视为明显异常，需专项分析。

### E. 偏黑 / 曝光 / 相机效果异常

优先判断是不是 Android 采集配置或设备适配：
- 对比原生 camera 画面与 SDK 采集画面
- 若 SDK 明显更黑：可让 QA 借机复现
- 若特定 `nertc.cfg` 配置可规避：说明更像设备配置/适配问题
- 若仍不可解：升级 SDK 开发

### F. 屏幕共享 / 特殊低性能设备

重点看：
- 共享内容类型：动画优先还是细节优先
- 共享帧率配置是否合理
- 软编/硬编是否符合预期
- 老旧设备是否性能不足

经验：
- 文档模式优先清晰度：建议较低帧率 + detail 模式
- 动画/视频模式优先流畅度：建议更高帧率 + motion 模式
- 低性能设备共享卡顿常与默认软编有关

## 证据不足时必须补充

以下情况必须追补：
- 无画面：补本地预览情况、是否设置远端画布、是否订阅远端视频
- 模糊：补是否大/小流、发送端分辨率、接收端分辨率、QP、码率、拉流地址或录屏
- 前处理嫌疑：补美颜/虚拟背景/鉴黄 SDK 供应商和版本
- 黑屏/偏黑：补原生相机截图 vs SDK 截图
- 共享问题：补共享配置、帧率、编码模式、设备性能档位

## 不要轻易下的结论

- 不要只因“模糊”就断言网络差；小流订阅和内容生产质量都可能是主因。
- 不要忽略 `setupRemoteVideoCanvas`；很多“有码流没画面”其实是渲染/画布问题。
- 不要把本地预览异常直接归因网络；本地预览不经过下行网络。
- 不要把所有黑屏都归因 SDK；前处理、设备占用、摄像头权限都很高频。
- 不要用单个版本 case 直接外推全部版本。

## 升级条件

- 绿块、码流错误、编解码异常征兆明显
- 指标都正常但主观异常稳定复现
- 特定机型/系统/第三方前处理稳定复现
- QP/码率/VQC 表现异常，需视频研发或网络研发介入
- 共享流/低端机硬编软编策略需要专项确认
