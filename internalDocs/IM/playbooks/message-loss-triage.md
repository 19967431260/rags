# Playbook：丢消息问题排查（点对点 & 群）

- source file: `/Users/yxgc/.openclaw/workspace/tmp/popo-doc-export/docs/272a99ba600a479e92b8d67d5418e6af__丢消息问题排查-点对点-群.md`
- 定位：**首轮分诊 playbook**。目标不是一步定责，而是先判断问题在发送、在线接收、离线补偿还是换端漫游。

## 适用场景

- 用户反馈“发了消息，对方没收到”
- 点对点或群消息都可先用本流程做入口分诊
- 已知大致发送时间、发送方/接收方，但不确定问题发生在哪一段链路

## 最小输入

至少要有：
- 发送方账号
- 接收方账号 / 群 ID
- 发送时间点（尽量精确到分钟）

强烈建议同时补充：
- clientId / serverId / 消息ID 任一
- 发送方日志
- 接收方日志
- 问题时接收方是否在线
- 是否为“当时没收到”还是“换设备后没看到”

## 首先判断

按下面顺序先做三分法：
1. **消息是否发送成功**
2. **若发送成功，接收方当时是否在线**
3. **若接收方离线或换端，问题是否已转为离线/漫游同步问题**

如果第 1 步都不能确认，就不要直接进入客户端或漫游结论。

---

## 分支排查路径

### A. 先确认发送是否成功

优先证据：
- 有 clientId / serverId：查发送协议 **7-1** 返回码是否为 `200`
- 没有消息 ID：用发送方、接收方、发送时间，查 **7-1** 或 QS 消息记录，看是否存在该条消息
- 有发送方日志：直接看 **7-1** 返回错误码

判断：
- **7-1 非 200 / 明确报错** → 优先归类为“发送未成功”，不是“服务端丢消息”
- **7-1 为 200 / 服务端有消息记录** → 进入接收链路排查
- **证据不足** → 先补消息标识或发送日志，不要跳结论

### B. 发送成功后，判断接收方是否在线

#### B1. 接收方在线但没收到

建议：优先拿接收方日志。

检查：
- 搜对应消息的 `clientId`
- 搜接收侧是否有 **4-3** 协议
  - 原文语义：在线收到消息时，接收端通常会马上发一条 **4-3**

判断：
- **日志看到消息已到端 / 有对应 4-3** → 更偏客户端后续处理或展示问题
- **发送成功，但接收方在线仍完全没收到** → 原文建议：优先找客户端开发看，**大概率是客户端入库失败**

此分支下不要轻易说：
- “服务器丢消息了”
- “一定是网络问题”

#### B2. 接收方离线，后续登录仍没收到

先确认该消息是否允许离线/漫游补偿：
- 查 **7-1** 或 **8-2**，确认消息是否设置了存离线/可补偿属性

然后：
- 直接转入 **漫游问题排查** playbook

这是因为：
- 离线后登录没收到，本质已不只是“实时投递”，而是“离线库/漫游库是否入库并同步”问题

#### B3. 接收方当时在线且收到了，但后面换设备登录没看到

判断：
- 该情况通常**不会再走离线库补发**，而是通过**漫游库**下发

然后：
- 直接转入 **漫游问题排查** playbook

## 证据不足时必须补充

缺任一关键证据时，都不建议定责：
- 精确发送时间
- 双方账号 / 群 ID
- 接收方当时在线状态
- clientId / serverId / 消息ID
- 发送方日志或接收方日志
- 问题现象是否发生在“当时”还是“换端后”

## 不要轻易下的结论

- 不要仅凭“用户说没收到”就判定服务端丢消息
- 不要在未核实 7-1 返回码前说消息已成功发出
- 不要把“换设备后没看到”误判成实时接收失败
- 不要在没有接收方日志时，直接把在线场景归因给服务端

## 升级条件

满足以下情况再升级更合理：
- 7-1 明确成功，且接收链路证据与服务端记录互相矛盾
- 离线/换端场景进入漫游排查后，发现漫游库、同步协议、客户端日志仍无法闭环
- 需要确认特殊场景：拉黑、第三方回调等（原文明确该部分尚未补齐）
