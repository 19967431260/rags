<!--keywords: 第三方回调, 回调,干预事件处理，事件回调 -->

网易云信的第三方回调功能，是专为企业打造的事件响应机制。简而言之，它允许企业应用服务器主动介入事件处理流程，以实现更精细的控制。通过第三方回调功能，企业不仅可以增强消息传递的可控性，还能深度整合自身的业务流程，实现与网易云信服务的无缝对接，从而更好地满足企业级通信和内容管理的需求。

<!--
通过回调机制，应用服务器可以拦截、检查和修改各种操作，如消息发送、群组管理等。可以应用在企业办公场景下的消息审核、接入第三方内容反垃圾服务以及自定义业务逻辑等场景中。
-->

## 技术原理

开通第三方回调服务后，网易云信 IM 服务器会在某一事件发生前后，以 HTTP/HTTPS 请求的方式发送给客户的应用服务器，应用服务器可以据此进行数据同步或干预事件的处理流程，参与决策网易云信的各类事件处理结果。<!--以单聊消息为例，在发送方发送消息后，IM 服务器不直接投递给接收方，而是先向应用服务器发送请求，将消息内容和用户信息发往应用服务器。用户的应用服务器判断消息是否可以发送，并返回结果，网易云信 IM 服务器再选择是否投递消息。-->

具体的流程步骤如下：

1. 网易云信服务器接收用户请求并进行初步校验。
2. 网易云信服务器向您的应用服务器发送回调请求，邀请应用服务器参与决策，进行业务判断并返回处理结果。
3. 云信服务器根据返回结果决定是否执行原始操作。

::: note notice
- 第三方回调服务仅针对通过 SDK 发送的消息。通过服务端 API 发送的消息，不会触发第三方回调。
- 如果已开通第三方回调并配置消息相关回调，即使用户被对方拉黑，发消息时仍会先进行第三方回调。
:::

## 开通和配置第三方回调

使用第三方回调前，您需要在 [网易云信控制台](https://app.yunxin.163.com/global/home) 上开通和配置 **第三方回调**。步骤详情请参考 [开通和配置第三方回调](https://doc.yunxin.163.com/console/concept/Tk3NjMzMjg?platform=console)。

## 回调请求规范

| 请求相关项 | 说明 |
| ---- | ---- |
| 请求协议 | HTTP/HTTPS，为保证数据安全，建议您使用 HTTPS 协议。 |
| 请求方式 | POST，后面接 **第三方回调地址**（您在 [网易云信控制台](https://app.yunxin.163.com/global/home) 配置的第三方回调地址，通常为您的应用服务器地址），例如：`POST https://******.com`。 |
| 消息格式 | `application/json; charset=utf-8` |
| 回调地址 | 对于部分回调类型（如单聊消息和群组消息）来说，您可在发送消息时设置环境变量（env），服务器将根据不同的环境回调到不同的回调地址，环境和回调地址的映射关系需要在 [网易云信控制台](https://app.yunxin.163.com/global/home) 配置，具体请参考 [开通和配置第三方回调](https://doc.yunxin.163.com/console/concept/Tk3NjMzMjg?platform=consol)。 |
| 校验方式 | 网易云信 IM 服务器请求您的应用服务器时，您的应用服务器从请求头 （request header）中获取 `CheckSum` 进行安全校验。 <li>`CheckSum` = sha1(`AppSecret` + `MD5` + `CurTime`)，其中 `AppSecret`、`MD5` 和 `CurTime` 均为 String 类型。<li>在验证数据是否在传输过程中被篡改时，需要计算验证 `MD5` 值是否被修改，以及计算验证 `CheckSum`。<li>`AppSecret` 值为您的 AppSecret（与 AppKey 对应），`MD5` 值为根据请求体（request body）计算的值。|
| 请求次数 | 网易云信 IM 服务器只会请求 1 次，无重试。 |
| 请求超时时间 | <p>超时时间为 2 秒。</p><li>如果请求失败或者超时，网易云信 IM 服务器会使用网易云信控制台上配置的默认回调结果继续处理业务逻辑。<li>您可前往 **产品功能 > IM 即时通讯 > 基础功能 > 第三方回调 > 子功能配置** 修改调用失败时的默认策略。 |
| 消息放行结果 | <li>如果回调不通过，会返回 `403`，表示非法操作或没有权限。<li>如果回调通过，会继续执行后续业务逻辑。<note type="note">当 SDK 在执行后续业务逻辑时遇到错误，会返回特定的错误码，这些错误码与回调机制无关。例如，如果某个操作或请求触发了黑名单机制，系统会返回一个特定的错误码 `7101`，表示被接收方加入黑名单。 |

## 请求格式

### 请求说明

请求头参数如下：

| Header 参数 | 类型 | 说明 |
| ---- | ---- | ---- |
| `AppKey` | String | 您的应用的 App Key，具体获取方式请参考 [获取 App Key](https://doc.yunxin.163.com/console/guide/TIzMDE4NTA?platform=console#获取-appkey-和-appsecret)。 |
| `CurTime` | Long | 当前 UTC 时间戳，从 1970 年 1 月 1 日 0 点 0 分 0 秒开始到现在的毫秒数（Long）。 |
| `MD5` | String | 根据请求中的 request body 计算出来的 MD5 值。 |
| `CheckSum` | String | 校验值。 |
| `Content-Type` | String | 请求消息体类型，一般为：`application/json`。 |

- MD5 值计算示例：

    ```Java
    String requestBody = "{}";
    String MD5 = CheckSumBuilder.getMD5(requestBody); //参考 接口概述 -> API checksum 校验部分
    ```
- CheckSum 值计算示例：

    ```Java
    String AppSecret = "90ud57s6****";
    String MD5 = "9894907e4ad9de467809127750******";
    String CurTime = "1440570500855"; ////当前 UTC 时间戳，从 1970 年 1 月 1 日 0 点 0 分 0 秒开始到现在的毫秒数
    String CheckSum = CheckSumBuilder.getCheckSum(AppSecret, MD5, CurTime); //参考 接口概述 -> API checksum 校验部分
    ```

请求体统一为 JSON 格式，示例：

```JSON
{
    "body": "Hello",
    "eventType": 1,
    "fromAccount": "000266",
    "fromClientType": "WEB",
    "fromDeviceId": "617715aa8579db03f0cf054c199c****",
    "fromNick": "yj000266",
    "msgTimestamp": "1541560157286",
    "msgType": "TEXT",
    "msgidClient": "",
    "to": "005877",
    "fromClientIp":"115.211.**.**", //ipv4 地址或 ipv6 地址
    "fromClientPort":"568**"
}

//JSON 中的属性请参考具体的回调类型
```

### 响应说明

第三方回调响应的 Content-Type Header 需要设置为 `application/json; charset=utf-8`。

响应体为 JSON 格式，示例：

```JSON
{
    "errCode":0,
    "responseCode": 20000,
    "modifyResponse": {},
    "callbackExt": "aa"
}
```

| 字段 | 说明 |
| ---- | ---- |
| `errCode` | <li>`0`：表示回调通过，允许执行。<li> `1`：表示回调不通过，取消执行。如果设置了合法的自定义错误码（`responseCode`），则发送端会收到自定义错误码，否则发送端会收到 `403` 错误码。 |
| `responseCode` | <li>当 `errCode` 为 `1` 时有效。<li>取值范围：`20,000` 至 `20,099`。<li>对于消息类型的第三方回调（`eventType`=`1`、`2`、`6`、`22`、`41`、`72`、`73`、`74`），支持设置为 `200` 的错误码，客户端表现为消息发送成功，其实消息发送失败。 |
| `modifyResponse` | <ul><li>仅对 IM & 圈组消息相关的回调类型有效（`eventType`=`1`、`2`、`6`、`22`、`41`、`72`、`73`、`74`），用于修改消息内容。JSON 格式，支持 `body`、`attach`、`ext`、`pushContent`、`pushPayload`、`pushEnable`、`needPushNick`、`persistEnable`、`skipHistory`（聊天室消息有效）、`roamingEnable`、`historyEnable` 等字段（均可选，若不填则不替换），上述字段的长度限制和正常发消息的限制相同。示例：`{"body":"xxx","attach":"xxx","ext":"123"}`。</li><li>仅对登录聊天室回调类型有效（`eventType`=`37`）。JSON 格式，支持 `tag`、`notifyTargetTag` 字段，进入聊天室时会修改目标用户的 `tag` 和 `notifyTargetTag`，`notifyTargetTag` 是 <a href="https://doc.commsease.com/messaging2/guide/TkxNTg3NTk?platform=client#核心概念" target="_blank">标签表达式</a>。示例：`{"tag":["tag1","tag2","tag3"],"notifyTargetTag":"{\"tag\":\"tag1\"} and {\"tag\":\"tag2\"}"}`。</li><li>圈组相关：<ul><li>圈组发送消息有效：`body`、`attach`、`ext`、`pushContent`、`pushPayload`、`pushEnable`、`needPushNick`、`historyEnable`。</li><li>圈组发送系统消息有效：`body`、`attach`、`ext`、`pushContent`、`pushPayload`、`pushEnable`、`needPushNick`。</li><li>圈组更新消息有效：`body`、`ext`、`msg`、`operatorExt`、`pushContent`、`pushPayload`。</li><li>圈组更新系统通知消息有效：`body`、`ext`、`msg`、`operatorExt`、`pushContent`、`pushPayload`。</li></ul></li><li>效果：消息接收方收到消息的上述几个字段将会被替换，消息发送方无感知，但是消息发送方的多端设备收到的消息是修改后的，此外，离线消息、漫游消息、云端历史消息，存储的均是修改后的消息内容，因此不管是消息发送方还是接收方，从网易云信服务器获取到的消息均是修改后的消息。</li></ul><note type=notice>该 JSON 体中的字段如果超过其长度限制，则该字段的修改无效。</note> |
| callbackExt | <li>仅对 IM 消息相关的回调类型有效（`eventType`=`1`、`2`、`6`、`22`），用于传递第三方回调的扩展信息，最大 1024 个字符。<li>消息发送者和消息接收者均能获取该扩展字段（需要 SDK 版本大于等于 v7.7.0）。<li>若 `errCode=1`，则只有消息发送者能获取到该字段。<note type="notice">只有服务端的 `responseCode` 返回 200 时，客户端才能读取 `callbackExt` 的内容。 |
